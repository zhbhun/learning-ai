# LLM 测试问题集合设计

## 背景

将附件中的 LLM 经典测试项整理到 `turotials/challenge/README.md`，形成既可直接复制执行、又能追溯到相关基准资源的问题集合。目标文件当前为空；不恢复其 Git 历史中的旧题库。

## 内容范围

- 保留附件中的 13 个分类：推理、数学、算法 Coding、软件工程、前端、Three.js / 3D、WebGL 图形、Agent、工具调用、长上下文、视觉、指令遵循和中文。
- 每个附件候选项对应一个条目，总计 39 个条目。
- 能独立运行的条目提供完整的代表性提示词。
- 依赖代码仓库、数据集、长文档、图片或外部工具的条目提供可复用的任务模板，并说明所需输入。
- 不加入参考答案、评分细则或已删除的旧版 README 内容。

## 文档结构

文档以“LLM 能力测试问题集”为标题，并提供简短使用说明。每个条目统一采用以下结构：

````md
## 分类

### 问题名称

```md
可直接复制的提示词或任务模板
```

- 来源/资源：[项目或论文名称](官方地址)；必要时补充使用说明。
````

## 编写规则

1. 优先使用中文提示词；代码、API 名称和项目专有名词保留原文。
2. 自拟题明确标注为“代表性测试题，能力维度参考某基准”，不冒充基准原题。
3. 基准题库只提供官方获取入口，不复制整套测试数据。
4. 外部输入使用语义明确的占位符，例如 `<代码仓库路径>`、`<待分析文档>` 和 `<图片>`；这些占位符是任务接口的一部分。
5. 来源优先采用官方项目主页、官方 GitHub 仓库或原始论文，去除无关的跟踪参数。
6. 提示词应明确交付物、关键约束和可验证的完成条件，避免只写测试名称。

## 验收标准

- 13 个分类、39 个候选项均有对应条目。
- 所有条目符合统一的 Markdown 层级和代码块格式。
- 每个条目至少包含一个可访问的来源或资源地址。
- 自拟代表题与官方基准题的身份标注清晰。
- 外部依赖型条目说明所需输入，不伪造可直接运行的上下文。
- Markdown 结构检查通过，官方链接经抽样或批量核验。
