# 工作流

用脚本确定性编排多个子代理。

## Workflow

执行工作流脚本,确定性地编排多个子代理。工作流在后台运行——此工具立即返回 task ID,完成时通过 `<task-notification>` 通知你;用 `/workflows` 查看实时进度。

**适用场景:** 工作流把工作结构化地分配给多个 agent,目的有三:全面(分解后并行覆盖)、确信(承诺前用独立视角和对抗检查)、规模(承担单上下文装不下的迁移、审计、广扫)。脚本就是你编码这一结构的地方。

**何时调用:** 仅当用户明确选择多代理编排时调用此工具。显式选择指以下情形之一:

- prompt 中含关键词 `"ultracode"`;
- 会话开启了 ultracode;
- 用户用自己的话直接要求跑工作流或多代理编排;
- 用户调用了其指令要求调用 Workflow 的技能或斜杠命令;
- 用户要求跑某个具名或已保存的工作流。

任何其他任务,即便明显受益,也不要调用。

**调用模式:** 正确做法通常是混合——先内联侦察(列文件、找通道、scope diff)发现工作列表,然后交给工作流管线处理。常见可跨轮链的单阶段工作流:Understand(并行读者 → 结构化地图)、Design(判官面板)、Review(找 → 对抗验证)、Research(多模态扫 → 深读 → 综合)、Migrate(发现站点 → 各转换 → 验证)。

**Ultracode 模式:** 一旦确认开启,该选择即常设——默认为每个实质任务编写并跑工作流。目标是产出尽可能详尽、正确的答案,token 成本不是约束。多阶段工作流通常意味着按顺序串联多个工作流。质量模式(对抗验证、视角多样验证、判官面板、loop-until-dry、多模态扫、完整性批评、无静默上限)是手段,按需挑选;倾向用工作流编排并对抗验证发现,除非工作琐碎或已验证;只有对话轮或琐碎机械编辑才 solo。

**脚本编写:**

- 通过 `script` 内联传入,不要先 Write 到文件。每次调用会自动把脚本持久化到会话目录;迭代时用 Write/Edit 编辑该文件,再用 `{scriptPath}` 重调。
- 每个脚本必须以 `export const meta = {...}` 开头(纯字面量,无计算值)。必填字段:`name`、`description`;可选:`whenToUse`、`phases`。
- 脚本体内可用的 hook:`agent()`、`pipeline()`(多阶段无屏障,默认)、`parallel()`(屏障,需要所有结果一起时才用)、`log()`、`phase()`、`args`、`budget`、`workflow()`。
- 子代理被告知其最终文本即返回值,因此应返回原始数据;结构化输出用 `schema` 选项,校验在工具调用层完成。
- 默认使用 `pipeline()`;仅当 stage N 确实需要 stage N-1 的全部跨项上下文时,才伸手用屏障(`parallel()`)。

**质量模式:** 对抗验证、视角多样验证、判官面板、loop-until-dry、多模态扫、完整性批评、无静默上限。按用户要求的规模扩展。

**恢复:** 工具结果含 `runId`。恢复用 `Workflow({scriptPath, resumeFromRunId})`——最长未改动的 `agent()` 前缀即时返回缓存结果;首个编辑或新调用及其之后全部实跑。

**参数:**

- `script`(必填,≤524288 字符)
- `scriptPath`
- `name`
- `args`
- `resumeFromRunId`
- `title`
- `description`(忽略)
