# 监控

后台监听脚本输出,逐行推送事件通知。

## Monitor

启动一个后台监听器,从一个长运行脚本流式推送事件。脚本的每一行 stdout 就是一个事件——你可以继续处理其他工作,通知会以聊天消息的形式到达。事件按自身的节奏送达,而非用户回复;即便你正在等待用户答题,事件也可能落地。

按你需要几个通知选择实现方式:

- **只需一个通知**(如"服务器何时就绪 / 构建何时完成"):改用 Bash 配合 `run_in_background`,跑一个条件满足即退出的命令,例如 `until grep -q "Ready in" dev.log; do sleep 0.5; done`。条件满足、命令退出时你得到单个完成通知。
- **每个事件都通知、持续不断**(如"每次出现 ERROR 行都告诉我"):用 Monitor 配一个无界命令(`tail -f`、`inotifywait -m`、`while true`)。
- **每个事件都通知、直到已知终点**(如"推送每个 CI 步骤结果,运行完成即停止"):用 Monitor 配一个先发事件后退出的命令。

你的脚本 stdout 即事件流,每行变为一条通知;脚本退出即结束监听。

**脚本质量**:管道的每个阶段都必须逐行 flush,否则匹配项会卡在缓冲区无法输出——`grep` 需加 `--line-buffered`,`awk` 需调用 `fflush()`;`head` 完全无法 flush,`| head -N` 在累积到 N 个匹配前不会交付任何内容,然后才结束流。轮询循环中要处理瞬态失败(如 `curl ... || true`),单次请求失败不应整段杀掉监听。轮询间隔建议:远程 API 30 秒以上,本地检查 0.5–1 秒。

**覆盖率——沉默不等于成功**:当你在监听某个结果时,过滤器必须匹配每一个终态,而不仅是 happy path。问自己:如果这个进程此刻崩溃,过滤器会不会发出任何东西？不会,就放宽匹配条件,宁可多带一些噪声,也不要错过崩溃循环。

**输出量控制**:每一行 stdout 都会变成一条对话消息,所以过滤器要精选——但"精选"指的是"你会采取行动的行",而不是"只看好消息"。绝不直接 pipe 原始日志;过滤到你真正关心的成功与失败信号即可。输出过多的监听器会被自动停止,此时应收紧过滤条件后重启。

stdout 中 200 毫秒内的多行会被合并成一条通知,因此来自单个事件的多行输出会自然成组。

脚本运行在与 Bash 相同的环境中。退出会结束监听(退出码会被报告);触发 `timeout` 会被杀掉。设置 `persistent: true` 用于会话级长监听(如 PR 监听、日志 tail),可用 TaskStop 提前取消。

**WebSocket 源**:传入 `ws` 可开启 WebSocket 流,每个入站文本帧即一个事件;二进制帧以占位行报告。无需 shell、无需轮询——服务器推送、你被通知。优先于 `websocat wss://…`,避免额外进程和行缓冲陷阱;只有当你需要用 shell 工具转换或过滤帧时,才改用 bash 实现。

参数:

- `description`(必填):该监听器的简短描述,会出现在每条通知里。
- `timeout_ms`(默认 300000,最大 3600000;`persistent: true` 时忽略):到点即杀监听。
- `persistent`(默认 false):设为 true 表示会话级长监听。
- `command`:要执行的 shell 命令或脚本,每行 stdout 即一个事件;退出即结束监听。
- `ws`:要打开的 WebSocket,含 `url` 与可选 `protocols`;每个文本帧即一个事件。与 `command` 互斥,不能同时提供。
